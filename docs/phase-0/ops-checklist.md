# Phase 0 — 보안·법무·운영 체크리스트 (패러독스 작성)

> 상용 운영 전제. 적대적 관점에서 우리가 빠뜨릴 수 있는 모든 함정.
> 항목당 (체크박스 / 왜 / 검증 / 책임자).

## 1. 인증·인가

- [ ] **JWT 만료시간 1시간 이하**
  - 왜: 토큰 도난 시 노출 시간 최소화
  - 검증: Supabase Dashboard → Auth → JWT settings
  - 책임: 미라지(설정) / 패러독스(검증)

- [ ] **Refresh token rotation ON**
  - 왜: 회전 안 하면 탈취된 refresh로 영구 세션 발급 가능
  - 검증: Auth settings → Refresh Token rotation 활성

- [ ] **Session 길이 정책 명시** (예: 30일 inactive 시 만료)
  - 왜: 잊혀진 세션이 영구 활성화되는 위험
  - 검증: Auth settings → Inactivity timeout

- [ ] **이메일 인증 강제** (이메일 가입 흐름)
  - 왜: 미인증 이메일로 스팸 계정 양산 방지
  - 검증: Auth settings → Email confirmations 활성, 인증 안 된 계정의 게시글 작성 차단 RLS

- [ ] **비밀번호 정책**: 최소 8자, 일반적인 leak 비밀번호 차단
  - 왜: 약한 비밀번호 + 공개 leak 결합 시 즉시 탈취
  - 검증: Auth settings → Password strength + HIBP API 연동 검토

- [ ] **카카오·네이버·구글 OAuth callback URL 화이트리스트 정확**
  - 왜: 잘못된 callback 등록 시 인가 코드 탈취 가능
  - 검증: 각 콘솔의 등록 URL과 Supabase Redirect URLs 비교, 정확 일치

- [ ] **OAuth state 파라미터 검증** (특히 네이버)
  - 왜: CSRF 방어. Supabase가 자동 처리하지만 커스텀 흐름 만들 때 누락 주의
  - 검증: 직접 콜백 핸들러 작성 시 state 검증 코드 리뷰

- [ ] **관리자 계정 분리**: 일반 사용 계정과 분리, MFA 필수
  - 왜: 평소 사용 계정 도난 = 관리자 권한 도난 방지
  - 검증: profiles.role='admin' 계정은 MFA 활성 확인 + 별도 이메일

- [ ] **관리자 화면 IP 제한 옵션 (Phase 2 게이트)**
  - 왜: 추가 방어 레이어
  - 검증: 미들웨어 또는 Cloudflare WAF 규칙

- [ ] **service role key는 절대 클라이언트 노출 금지**
  - 왜: 노출 시 전체 DB RLS 우회 가능
  - 검증: 빌드 산출물 `NEXT_PUBLIC_*` 환경변수 grep, .env.local은 .gitignore, Vercel env는 Encrypted

- [ ] **JWT 비밀(JWT Secret) 정기 회전 정책 수립**
  - 왜: 키 유출 대비
  - 검증: 분기 1회 또는 사고 시 회전 절차 문서화

- [ ] **세션 쿠키: HttpOnly, Secure, SameSite=Lax**
  - 왜: XSS·CSRF 부분 방어
  - 검증: 브라우저 dev tools에서 쿠키 속성 확인, Supabase ssr 설정 검증

- [ ] **로그인 실패 5회 계정 임시 잠금 (15분)**
  - 왜: brute force 방어
  - 검증: 부하 테스트 5회 실패 후 6회째 거부

- [ ] **이메일 변경 시 양쪽 이메일에 알림 발송**
  - 왜: 계정 탈취 후 이메일 변경으로 복구 차단 시도 방어
  - 검증: 변경 흐름 테스트

- [ ] **비밀번호 재설정 토큰 1회용 + 1시간 만료**
  - 왜: 재사용 공격 방지
  - 검증: 동일 토큰 2회 사용 거부 테스트

- [ ] **소셜 OAuth 최소 권한 스코프**
  - 왜: 과도한 권한 = 사고 시 피해 ↑
  - 검증: 카카오: profile_nickname, account_email만. 네이버: 닉네임, 이메일만. 구글: openid email profile

- [ ] **로그아웃 시 모든 디바이스 옵션 제공**
  - 왜: 토큰 탈취 의심 시 즉시 무효화
  - 검증: signOut({ scope: 'global' }) 호출 동작

- [ ] **계정 삭제 흐름 — 확인 모달 + 비밀번호·OAuth 재인증**
  - 왜: 실수·악의적 삭제 방지
  - 검증: 삭제 흐름 수동 테스트

## 2. RLS·DB 보안

- [ ] **모든 public 스키마 테이블에 RLS 활성**
  - 왜: 누락 = 익명 사용자가 전체 데이터 read/write 가능
  - 검증: CI에서 `SELECT tablename FROM pg_tables WHERE schemaname='public' AND rowsecurity=false` → 결과 0 row 강제

- [ ] **각 테이블에 최소 1개 select 정책 + insert/update/delete 정책 존재**
  - 왜: RLS 활성만으로는 fail-closed. 정책 없으면 전부 거부 (이게 안전한데, 의도된 데이터 흐름이 막힐 수 있음)
  - 검증: pg_policies 조회로 테이블별 정책 수 확인

- [ ] **`SECURITY DEFINER` 함수에는 `SET search_path = public, pg_temp` 명시**
  - 왜: search_path 인젝션 공격 방어
  - 검증: 모든 SECURITY DEFINER 함수 SQL 리뷰

- [ ] **`SECURITY DEFINER` 함수의 입력은 명시적 캐스팅 + 검증**
  - 왜: RLS 우회 + 권한 상승 위험
  - 검증: has_role(), handle_new_user() 등 함수 input 정규화

- [ ] **anon role의 INSERT/UPDATE/DELETE 정책은 명시적 제로 또는 매우 좁게**
  - 왜: 익명 사용자가 데이터 수정 가능하면 즉시 사고
  - 검증: 모든 정책에서 `auth.uid() IS NOT NULL` 조건 확인

- [ ] **N+1 쿼리 + RLS 결합 성능 함정 모니터링**
  - 왜: RLS가 매 row 평가 → 1000개 row = 1000번 평가
  - 검증: EXPLAIN ANALYZE로 핵심 쿼리 검증, Slow query log

- [ ] **RLS 정책 사용 컬럼에 인덱스 보장**
  - 왜: 인덱스 없으면 시퀀스 스캔 + RLS 평가 폭증
  - 검증: author_id, status, is_deleted, role 등에 인덱스

- [ ] **마이그레이션에서 신규 테이블 RLS 누락 방지 lint**
  - 왜: 사람이 잊을 수 있음
  - 검증: pre-commit hook 또는 CI에서 `ENABLE ROW LEVEL SECURITY` 누락 검출

- [ ] **백업 자동 (Supabase PITR) + 별도 외부 백업 주 1회**
  - 왜: Supabase 자체 사고 시 복구 불가
  - 검증: pg_dump 자동 스크립트, 분기 1회 복원 리허설 실시

- [ ] **백업 파일 암호화 + 별도 저장소**
  - 왜: 백업 유출 = 전체 사용자 데이터 유출
  - 검증: gpg 암호화 + B2/S3 분리 버킷

- [ ] **DB 직접 접속 (psql) 권한 — 운영진 한 명에게만**
  - 왜: 직접 접속이 RLS를 완전히 우회
  - 검증: Supabase 접근 권한 명단 검토

- [ ] **prod DB 접속 시 audit log 필수**
  - 왜: 누가 무엇을 했는지 추적
  - 검증: Supabase Audit Logs 활성 (가능한 플랜)

- [ ] **개발/스테이징/프로덕션 DB 완전 분리**
  - 왜: dev 사고가 prod에 닿으면 치명적
  - 검증: 별도 Supabase 프로젝트 3개, 환경변수 명확 분리

- [ ] **prod 데이터의 dev 환경 복사 시 PII 마스킹 의무**
  - 왜: 개발자가 실 사용자 데이터 접근 위험
  - 검증: 마스킹 스크립트 + 정책 문서화

## 3. 입력 검증·OWASP Top 10

- [ ] **사용자 마크다운 → HTML 렌더링은 서버 사이드 DOMPurify**
  - 왜: 클라이언트만 사니타이즈 시 우회 가능
  - 검증: 마크다운 처리 코드 리뷰 + XSS 페이로드 테스트 (`<img src=x onerror=alert(1)>`)

- [ ] **자동링크 처리 시 `javascript:` `data:` 스킴 차단**
  - 왜: XSS 공격 벡터
  - 검증: 페이로드 테스트

- [ ] **rich text → body_html 캐시는 사니타이즈된 결과만 저장**
  - 왜: 캐시 자체가 공격 vector가 되지 않게
  - 검증: 코드 리뷰

- [ ] **SQL 인젝션 — raw SQL 직접 작성 금지, supabase-js 빌더만**
  - 왜: 스트링 concat은 항상 인젝션 위험
  - 검증: grep으로 `pool.query(\`...\${`` 등 패턴 검출, 코드 리뷰

- [ ] **rpc 함수 호출 시 인자는 supabase-js 파라미터 바인딩**
  - 왜: 동상
  - 검증: 코드 리뷰

- [ ] **CSRF — state-changing 작업은 토큰 기반 인증 + Origin 헤더 검증**
  - 왜: Supabase가 토큰 기반이라 부분 면역이지만 추가 액션(이메일 변경, 탈퇴)에는 명시적 토큰
  - 검증: 민감 액션 코드 리뷰

- [ ] **SSRF — 사용자 입력 URL을 서버가 fetch하는 자리(예: oEmbed) 허용리스트**
  - 왜: 내부 네트워크 스캔·메타데이터 서버 접근
  - 검증: URL 파서로 호스트 검증, 사설 IP·메타데이터 IP 차단 (169.254.169.254 등)

- [ ] **파일 업로드 — magic byte 기반 MIME 검증**
  - 왜: 확장자만 보면 우회 가능
  - 검증: 서버 사이드 file-type 라이브러리 사용

- [ ] **파일 업로드 — 크기 제한 10MB 이하**
  - 왜: 스토리지 비용·DOS 방어
  - 검증: Server Action에서 검증 + Supabase Storage 버킷 정책

- [ ] **이미지 EXIF 제거**
  - 왜: GPS·기기 정보 = 개인정보 + 위치 노출
  - 검증: 클라이언트 측 제거 + 서버에서 재검증

- [ ] **업로드 파일은 별도 서브도메인에서 서빙 또는 Content-Disposition: attachment** (HTML/JS 업로드 방어)
  - 왜: HTML 업로드 후 동일 도메인에서 실행 = XSS
  - 검증: Storage 응답 헤더 확인

- [ ] **IDOR — 본인 데이터만 수정/삭제. RLS가 첫 방어선**
  - 왜: URL 파라미터 변경으로 타인 데이터 접근 방지
  - 검증: RLS 정책 리뷰 + e2e 테스트로 타인 자원 시도 시 403

- [ ] **오픈 리다이렉트 — 로그인 후 returnTo는 same-origin 화이트리스트만**
  - 왜: 피싱 사이트로 리다이렉트 위험
  - 검증: returnTo 파서가 외부 호스트 거부 확인

- [ ] **CSP (Content Security Policy) 헤더 설정**
  - 왜: XSS 다층 방어
  - 검증: default-src 'self', script-src 'self' 'nonce-...', img-src 자체+Supabase Storage 도메인, frame-src YouTube 등 명시

- [ ] **HSTS 헤더**
  - 왜: SSL stripping 방어
  - 검증: `Strict-Transport-Security: max-age=63072000; includeSubDomains; preload`

- [ ] **X-Frame-Options: DENY** (관리자 화면 + 일반 페이지)
  - 왜: clickjacking 방어
  - 검증: 응답 헤더 확인

- [ ] **CORS — supabase 외 외부 origin 차단**
  - 왜: 다른 사이트에서 API 직접 호출 차단
  - 검증: CORS 헤더 응답 확인

  - 왜: 작성자 권리, 항의 채널

  - 왜: 명예훼손·도용 등은 시간 민감
